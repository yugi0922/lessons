# 1. CI/CDが攻撃者にとって価値の高い理由

発表者メモ：

受講者への問いかけ

「攻撃者だとしたら、開発者1人のPCとCI/CDサーバーならどちらを狙いますか？」

---

## 1.1 CI/CD環境に集まるもの

### 原稿

まず最初に、

なぜ攻撃者がCI/CDを狙うのかを考えてみます。

CI/CDには大きく4種類の重要なものが集まっています。

1つ目は権限です。

* リポジトリ書き込み
* パッケージ公開
* クラウドへのデプロイ
* コンテナレジストリへのPush

などです。

2つ目は認証情報です。

* npm Token
* AWS Credential
* GitHub PAT
* Signing Key

などがあります。

3つ目は成果物です。

* Build Artifact
* Dependency Cache
* Build Cache

などです。

4つ目は自動化です。

PushやPRをきっかけに自動で実行され、

人が毎回確認するわけではありません。

つまり、

強い権限と認証情報が集中していて、

しかも自動で動く。

これが攻撃者にとって魅力的な理由です。

---

## 1.2 攻撃者の視点

### 原稿

資料では5つの理由が挙げられています。

* 権限の集中
* 自動実行
* 信頼の連鎖
* 持続性
* 横展開

です。

CI/CDを侵害できれば、

本番環境やクラウドへアクセスできる可能性があります。

また、

CIで作られた成果物は信頼されやすいため、

攻撃者にとって非常に効率の良い攻撃対象になります。

### この章のまとめ

CI/CDは単なる自動化ツールではありません。

権限とSecretsが集中するため、

攻撃者にとって価値の高い標的です。

---

# 2. ライブラリ汚染とCI/CDの関係

## 章へのつなぎ

ここまでで、

なぜCI/CDが狙われるのかを見てきました。

では実際に、

悪意あるライブラリが混入すると、

どのように被害が広がるのでしょうか。

次はその流れを見ていきます。

---

## 2.1 被害拡大の流れ

### 原稿

この図は今回の資料の中でも重要な図です。

流れを順番に見ていきます。

まず、

悪意あるライブラリがlockfileへ入ります。

その後、

PRがマージされます。

するとCI/CDが起動し、

npm ciやgradle buildが実行されます。

ここで重要なのは、

悪意あるコードが実行されることです。

攻撃者は、

環境変数やSecretsを取得しようとします。

例えば、

* AWS Credential
* NPM Token
* Deploy Credential

などです。

そして盗んだ認証情報を利用して、

本番環境へのアクセスやパッケージ改ざんを行います。

つまり、

ライブラリ汚染だけでは終わりません。

CI/CD上で実行されることで被害が拡大します。

---

## 2.2 分離されていない場合の問題

### 原稿

資料のWorkflowを見てください。

一つのJobの中で、

* npm ci
* test
* build
* publish
* deploy

をすべて実行しています。

一見問題なさそうですが、

ここに大きな落とし穴があります。

PublishやDeployで利用するSecretsは、

Job全体から利用できます。

つまり、

npm ciのタイミングでも利用可能です。

もし悪意あるライブラリが実行された場合、

その時点でSecretsが盗まれる可能性があります。

### この章のまとめ

ライブラリ汚染だけではなく、

CI/CD実行で被害が拡大します。

# 3. GitHub Actions workflowの permissions

## 章へのつなぎ

前の章では、

Secretsが狙われる流れを見ました。

そこで次は、

GitHub Actionsの権限をどう制御するのかを見ていきます。

---

## 3.1 デフォルトのpermissions

### 原稿

GitHub Actionsでは、

GITHUB_TOKENが自動発行されます。

そのため、

何も設定していなくても権限を持っています。

資料では、

デフォルトPermissionsをread-onlyにすることを推奨しています。

考え方はシンプルです。

最初から強い権限を与えず、

必要になったら追加する。

これが最小権限の原則です。

---

## 3.2 workflowレベルでpermissionsを明示する

### 原稿

Workflowでは、

permissionsを明示します。

例えば

contents: read

だけを許可します。

そうすることで、

レビュー時にも

「何を許可しているのか」

が明確になります。

---

## 3.3 主要なpermissions

### 原稿

ここで重要なのは、

一覧を覚えることではありません。

必要なものだけ付ける。

不要な権限は付けない。

という考え方です。

---

## 3.4 jobレベルでpermissionsを分ける

### 原稿

Test JobとDeploy Jobでは役割が違います。

そのため、

Permissionsも違ってよいのです。

例えば、

Deploy JobだけにOIDC用の権限を与えます。

これによって、

侵害されたとしても被害を限定できます。

### この章のまとめ

WorkflowやJobには、

必要最小限のPermissionsだけを与えます。

---

# 4. deploy jobとtest jobの分離

発表者メモ：

この資料で最重要の章。

必ず

「Secretsを持つJobでnpm ciしない」

を強調する。

---

## 4.1 分離の原則

### 原稿

ここが今回の資料で最も重要なポイントです。

依存ライブラリをインストールするJobと、

DeployするJobを分離します。

Test Jobでは、

npm ciを実行します。

しかし、

Secretsは持ちません。

Deploy Jobでは、

Secretsを持ちます。

しかし、

npm ciは実行しません。

つまり、

信頼できないコードと強い権限を分離します。

これがJob分離の目的です。

# 5. secretsの管理

## 章へのつなぎ

ここまでで、

Job分離によって

「Secretsを持つJobで依存ライブラリを実行しない」

ことが重要だと分かりました。

次は、

そのSecrets自体をどのように管理するのかを見ていきます。

---

## 5.1 secretsのスコープ

### Markdown

| スコープ                 | 設定場所         | アクセス範囲       |
| -------------------- | ------------ | ------------ |
| Organization Secrets | Organization | 指定Repository |
| Repository Secrets   | Repository   | Repository全体 |
| Environment Secrets  | Environment  | 指定Jobのみ      |

### 原稿

GitHub Actionsでは、

Secretsを保存する場所を選ぶことができます。

一番広い範囲が

Organization Secretsです。

複数Repositoryで共有できます。

便利ですが、

影響範囲も大きくなります。

次がRepository Secretsです。

そのRepositoryだけで利用できます。

そして最後がEnvironment Secretsです。

例えば、

Production用AWS Credentialを

Production Environmentだけで利用できます。

ここで覚えてほしいことは一つです。

Secretsは

**最も狭いスコープで管理する**

ということです。

Deploy Jobだけで利用するなら、

Repository全体から見える必要はありません。

Environment Secretsで十分です。

---

## 5.2 不要なSecretsの排除

### 原稿

Secretsは一度登録すると、

長期間残り続けることがあります。

しかし、

使われていないSecretsはリスクになります。

資料では、

* 実際に利用されているか
* 最小権限か
* 長期間更新されていないか
* 退職者が作成したものではないか
* 重複がないか

を確認しています。

これはシステム運用でも同じです。

不要な権限は削除する。

不要な認証情報も削除する。

定期的な棚卸しが重要です。

---

## 5.3 Secretsの露出防止

### 原稿

GitHubにはSecrets Maskingがあります。

ログへ出力しても、

値を隠してくれます。

しかし、

ここで誤解してはいけません。

Maskingは

ログを見えなくするだけです。

悪意あるライブラリが

外部サーバへ送信することまでは防げません。

つまり、

本当の対策はMaskingではありません。

Secretsを持つJobで

信頼できないコードを実行しないことです。

ここでもJob分離の考え方につながります。

---

### この章のまとめ

Secretsは最小スコープで管理します。

そして、

Secrets Maskingに依存するのではなく、

Secretsを持つJobで依存ライブラリを実行しない設計を行います。

---

# 6. fork PRとpull_request_targetのリスク

## 章へのつなぎ

ここまではSecrets管理について説明しました。

次は、

GitHub Actionsで実際によく問題になる

pull_request_targetについて説明します。

---

## 6.1 fork PRからのSecretアクセス

### 原稿

通常の

pull_request

では、

Fork Repositoryから送られたPRは

Secretsへアクセスできません。

これはGitHubが安全側に設計しているためです。

もしアクセスできると、

誰でもPRを送るだけで

認証情報を盗めてしまいます。

そのため、

通常のPRではSecretsは利用できません。

---

## 6.2 pull_request_targetの危険性

### 原稿

ここで登場するのが

pull_request_target

です。

このイベントは、

PRへのコメントやラベル付けなどで利用されます。

しかし、

通常のpull_requestとは違い、

Base Repository側の権限を利用できます。

つまり、

Secretsへアクセスできます。

ここで問題になるのが、

PR側のコードをCheckoutしてしまうケースです。

その状態で

npm ci

を実行するとどうなるでしょうか。

攻撃者が用意したコードが実行されます。

さらに、

Secretsも利用可能です。

つまり、

未信頼コードとSecretsが同じ場所に存在します。

これは非常に危険です。

---

### この章のまとめ

pull_request_targetは便利ですが、

未信頼コードを実行してはいけません。

レビュー時に見つけたら、

必ず確認するようにしてください。

---

# 7. third-party actionsの管理

## 章へのつなぎ

これまではアプリケーションの依存関係を見てきました。

GitHub Actions自体も依存関係を持っています。

---

## 7.1 Third Party Actionのリスク

### 原稿

皆さんは

actions/checkout

などを利用していると思います。

しかし、

これらも外部コードです。

つまり、

サプライチェーンの一部です。

資料にもあるように、

2025年には

tj-actions/changed-files事件

が発生しました。

Action自体が侵害され、

Secrets漏えいにつながりました。

つまり、

ライブラリだけではなく、

GitHub Actionも管理対象です。

---

## 7.2 Commit SHA固定

### 原稿

最も推奨される方法が

Commit SHA固定です。

例えば、

@main

は避けます。

なぜなら、

明日には中身が変わるかもしれないからです。

Tag固定よりも、

Commit SHA固定の方が安全です。

レビュー時にも、

どのコードが実行されるのか明確になります。

---

## 7.3 Dependabotによる管理

### 原稿

ただし、

SHA固定にも課題があります。

更新されなくなることです。

そのため、

Dependabotで継続的に更新管理します。

アプリケーションだけではなく、

GitHub Actionsも依存関係として管理します。

---

### この章のまとめ

Actionも依存関係です。

信頼できるActionを利用し、

SHA固定とDependabotで管理します。

# 8. OIDCによる長期Token削減

## 章へのつなぎ

ここまで、

Secretsを守る方法について説明してきました。

しかし、

そもそもSecretsを減らせるなら、

その方が安全です。

そこでOIDCを見ていきます。

---

## 8.1 長期Tokenの問題

### 原稿

従来のCI/CDでは、

AWS Access Keyなどを

GitHub Secretsへ保存していました。

しかし、

長期Tokenには問題があります。

もし漏えいすると、

失効するまで利用されます。

また、

定期的なローテーションも必要です。

つまり、

管理コストと漏えいリスクがあります。

---

## 8.2 OIDCによる解決

### 原稿

OIDCでは、

長期Tokenを保存しません。

Workflow実行時に、

GitHubがOIDC Tokenを発行します。

Cloud Providerはそれを検証し、

短期間だけ有効なCredentialを発行します。

つまり、

AWS Keyを保存する必要がありません。

---

## 8.3 OIDCの利点

### 原稿

OIDCの最大のメリットは、

Credentialが短命であることです。

仮に漏えいしても、

すぐに利用できなくなります。

さらに、

ローテーション作業も不要になります。

現在は、

長期TokenよりOIDCが推奨されています。

---

## 8.4 npm Trusted Publishing

### 原稿

同じ考え方はnpmにもあります。

npm Trusted Publishingでは、

npm Publish Tokenを保存しません。

GitHub Actionsから直接認証します。

これも、

長期Token削減の代表例です。

---

### この章のまとめ

長期Tokenを減らし、

短命Credentialへ移行する。

これがOIDCの目的です。

# 9. dependency cacheとbuild cacheの安全性

## 章へのつなぎ

ここまでで、

Permissions、

Job分離、

Secrets、

OIDCについて説明してきました。

次は見落とされがちなCacheについて見ていきます。

---

## 9.1 Cacheのリスク

### 原稿

CI/CDでは、

ビルド高速化のためにCacheを利用します。

例えば、

npm packageのダウンロード結果を保存します。

これは便利ですが、

攻撃者にとっても便利です。

資料のシナリオを見てください。

まず、

悪意あるライブラリが一度インストールされます。

その内容がCacheへ保存されます。

その後、

lockfileを修正したとしても、

Cacheが再利用される可能性があります。

つまり、

プロジェクトから削除したつもりでも、

Cache経由で残り続けることがあります。

---

## 9.2 Cacheの管理

### 原稿

対策は3つあります。

まず、

lockfileのHashをCache Keyへ含めることです。

依存関係が変わったら、

新しいCacheを利用します。

次に、

インシデント発生時はCacheも削除します。

ソースコード修正だけでは不十分です。

最後に、

Cache Poisoningのリスクを理解します。

Cacheも攻撃対象であることを忘れないでください。

---

### この章のまとめ

Cacheは単なる高速化機能ではありません。

セキュリティ管理の対象です。

# 10. リリース経路の保護

## 章へのつなぎ

ここまで個別の対策を見てきました。

しかし、

攻撃者は特定の場所だけを狙うわけではありません。

そこで、

リリース経路全体を見ていきます。

---

## 10.1 リリース経路とは

### 原稿

リリース経路とは、

ソースコードが利用者へ届くまでの流れです。

例えばWebアプリなら、

Source

↓

Build

↓

Test

↓

Staging

↓

Production

です。

npmなら、

最後はPublishになります。

攻撃者は、

Buildだけ、

Deployだけ、

を狙うとは限りません。

そのため、

リリース経路全体で考える必要があります。

---

## 10.2 保護ポイント

### 原稿

資料では、

各段階の対策が整理されています。

ソースコードでは、

Branch Protection。

Buildでは、

Dependency Verification。

Testでは、

自動テスト。

Deployでは、

OIDC。

成果物では、

Artifact Signing。

という形です。

重要なのは、

どれか一つで守るのではなく、

複数の防御を重ねることです。

これをDefense in Depthと呼びます。

---

## 10.3 Branch Protection

### 原稿

Branch Protectionは、

GitHubを利用するチームでは必須に近い機能です。

資料では、

* mainへの直接Push禁止
* Required Review
* CI成功必須
* Force Push禁止

が紹介されています。

これによって、

悪意ある変更が本番へ到達しにくくなります。

---

### この章のまとめ

個別の設定だけでは不十分です。

リリース経路全体を守ることが重要です。

## Q. Commit SHA固定なら100%安全ですか？

A.

いいえ。

信頼できるAction選定と、

Dependabotによる継続管理も必要です。

---

## Q. OIDCを使えばSecretsは不要ですか？

A.

すべて不要になるわけではありません。

Cloud認証情報は削減できますが、

アプリケーション固有のAPIキーなどは残る場合があります。

---

## Q. この資料で一番重要なポイントは何ですか？

A.

「信頼できないコードと強い権限を分離すること」です。

Job分離も、

pull_request_targetも、

OIDCも、

すべてこの考え方で説明できます。
